Don't Waste BRAM: FPGA-Aware KV Cache Quantization

Research progress report — 2026-04-19 | Phases 1–4 complete on SmolLM-360M

Abstract
Prior work (DWB, arXiv:2604.04722) allocates 47.9% of KV tokens to 2-bit quantization despite 2-bit having identical FPGA hardware cost as 4-bit on Xilinx Ultrascale+ BRAM. We propose a binary {4,8}-bit FPGA-aware controller that removes this suboptimal option. Our controller achieves 3.48× FPGA throughput vs. DWB's 2.44× (+43%) at equal accuracy (41.0% vs. 41.2%) with 21% fewer bits. We additionally characterize a scale-dependent losslessness threshold that explains when INT4 is unconditionally optimal.

The Core Insight: 2-bit Wastes No BRAM

Xilinx Ultrascale+ BRAM has a minimum port width of 4 bits. Any quantization below 4 bits still occupies a 4-bit BRAM slot:

BitsBRAM Port AccessesNormalized CostHellaSwag Acc
2-bit10.290 (same as 4-bit!)25.0% (catastrophic)
4-bit10.29041.6% (lossless)
8-bit20.56042.0%
16-bit (FP16)41.01042.6%
💡 The fundamental flaw in paper DWB: DWB allocates 47.9% of tokens to 2-bit. On Xilinx Ultrascale+ BRAM, 2-bit uses the same 4-bit port — no bandwidth saving. Yet 2-bit reduces accuracy from 41.6% → 25.0% (−16.6pp). Paper DWB's "2-bit savings" save zero FPGA bandwidth while destroying accuracy.

Paper DWB's FPGA cost (derived from their bit distribution: 47.9% 2-bit, 25.3% 4-bit, 15.4% 8-bit, 11.4% 16-bit):

FPGA_cost = 0.479×0.29 + 0.253×0.29 + 0.154×0.56 + 0.114×1.01 = 0.414 Speedup = 1.010 / 0.414 = 2.44×

Our binary {4,8} controller (100% 4-bit at 360M):

FPGA_cost = 1.00×0.29 = 0.290 Speedup = 1.010 / 0.290 = 3.48× (+43% over DWB)

Scale-Dependent Losslessness Theory

We characterize when INT4 is lossless using effective quantization residual:

ε_eff = ε_rel × (1 - r_cancel)
Modelε_relr_cancelε_effLossless?Acc (FP16 vs INT4)
SmolLM-135M0.2490.320.069✓ Yes40.0% vs 39.0% (−1pp)
SmolLM-360M0.2700.300.081✓ Yes42.6% vs 41.6% (−1pp)
SmolLM-1.7B0.3530.3510.124✗ No49.0% vs 41.1% (−7.9pp)

Losslessness threshold: between 8.1% and 12.4% effective residual. The 1.7B model's larger hidden dimension (2048 vs 960) produces higher KV variance → INT4 is genuinely lossy there.

Implication for the FPGA controller:
At 360M: Any controller correctly converges to 100% 4-bit (global optimum). FPGA advantage comes entirely from not allocating 2-bit tokens like DWB does.

At 1.7B: INT4 is lossy → controller learns mixed {4,8}-bit allocation → maximum benefit of binary FPGA controller. This is the key pending experiment (needs GPU).

Binary FPGA Controller Architecture

Simplified 3-layer MLP with Gumbel-softmax output:

Input: [kv_norm_normalized, position_fraction]  ← 2D per token
  → Linear(2, 64) → ReLU
  → Linear(64, 64) → ReLU
  → Linear(64, 2)  ← logits over {4-bit, 8-bit}
  → GumbelSoftmax(τ: 2.0 → 0.1)  ← differentiable bit selection
Output: soft probabilities p_4, p_8

FPGA-aware compound loss:

L = α × (1 - E[p·quality]) + β × (E[p·bram_cost] / 1.01) quality = [0.943, 0.966] ← normalized from HellaSwag empirical accuracy bram_cost = [0.290, 0.560] ← Xilinx Ultrascale+ BRAM port costs Both terms normalized to [0,1] — critical for gradient stability

Why normalization matters: Prior attempts with unnormalized quality (~1e-4 MSE scale) caused the compression term to dominate → 100% 2-bit collapse, 32% accuracy. Loss normalization is essential for stable training.
